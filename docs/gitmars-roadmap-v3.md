## 一、核心理念

### 1.1 设计初衷

```
┌─────────────────────────────────────────────────────────────┐
│                     Gitmars 的使命                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   让每个开发者都能轻松驾驭 Git                               │
│                                                             │
│   • 降低学习成本 —— 不懂 Git 原理也能高效协作                │
│   • 减少操作步骤 —— 复杂流程一键完成                         │
│   • 避免人为错误 —— 最佳实践内置，想错都难                   │
│   • 提升团队效率 —— 统一工作流，减少沟通成本                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 核心差异化

> **让用户不需要思考 Git**

```
Git 原生：用户需要知道用什么命令
Gitmars：工具告诉用户应该做什么
```

### 1.3 开发原则

| 原则 | 说明 |
|------|------|
| **务实推进** | 先做好基础，再追求创新 |
| **用户导向** | 每个功能都要解决真实痛点 |
| **保持简单** | 功能可以强大，但使用必须简单 |
| **小步快跑** | 每 4-6 周发布一个聚焦版本 |
| **长期主义** | 不追求短期功能堆砌，注重长期价值 |

## 四、v8.0.0 - 差异化功能版本

**发布目标**：提供独特价值，让用户"离不开"

**周期**：4-5 周

### 4.1 核心功能

#### 功能一：Worktree 深度整合（核心差异化）

**战略意义**：
- 解决真实痛点（多任务并行开发）
- 与 git 原生命令形成差异化
- 为后续 IDE 集成打基础

**分阶段实现**：

**阶段一：基础功能（P0，2d）**

```shell
gitm worktree list              # 列出 worktree（带状态）
gitm worktree create <branch>   # 创建 worktree（自动路径）
gitm worktree prune             # 清理无效 worktree
```

**阶段二：高级功能（P1，1.5d）**

```shell
gitm worktree create <branch> --link-modules  # 链接 node_modules
gitm worktree create <branch> --open          # 自动打开 IDE
gitm worktree sync                             # 同步配置
```

**阶段三：工作流整合（P1，1d）**

```shell
gitm start feature/login --worktree --open
gitm end feature/login  # 自动清理 worktree
```

**工作量**：4.5 天

#### 功能二：冲突预检系统（P0）

**价值**：合并前知道风险，避免"合完才发现冲突"

```shell
gitm check              # 全面检查
gitm check --combine    # 合并前检查
gitm check --update     # 更新前检查
```

**输出示例**：

```

├─────────────────────────────────────────────────────────────┤

│  潜在冲突文件:                                               │

│                                                             │
│  建议: 先与团队成员沟通后再合并                               │
```

### 5.3 功能一：智能建议系统（核心）

#### 5.3.1 功能概述

**设计理念**：被动提醒，不打扰用户

```shell
# 入口：直接输入 gitm
gitm

> 正在分析当前状态...

┌─────────────────────────────────────────────────────────────┐
│  📊 当前状态                                                  │
│  分支: feature/login  状态: 开发中                            │
│  领先: 3 个提交  落后: 1 个提交  未推送: 3 个提交              │
├─────────────────────────────────────────────────────────────┤
│  💡 建议操作                                                  │
│                                                             │
│  [1] ⚠️  同步上游代码                                        │
│      您的分支落后 develop 1 个提交                            │
│      建议: gitm update                                       │
│                                                             │
│  [2] 💡 可以合并到测试环境                                    │
│      您有 3 个提交可以合并                                    │
│      建议: gitm combine -d                                   │
│                                                             │
│  [3] 📝 推送本地提交                                          │
│      您有 3 个提交未推送                                      │
│      建议: git push                                          │
│                                                             │
├─────────────────────────────────────────────────────────────┤
│  输入数字执行对应操作，或按 [q] 退出                          │
└─────────────────────────────────────────────────────────────┘
```

#### 5.3.2 建议规则引擎

**规则定义**：

```typescript
interface SuggestionRule {
    id: string                    // 规则 ID
    name: string                  // 规则名称
    priority: 'high' | 'medium' | 'low'  // 优先级
    condition: (ctx: GitContext) => boolean | number  // 触发条件
    message: string               // 提示消息
    action: string                // 建议操作
    autoFix?: boolean             // 是否可自动执行
    category: 'warning' | 'info' | 'suggestion'  // 类型
}
```

**规则优先级排序**：

| 优先级 | 规则 ID | 触发条件 | 提示消息 | 建议操作 |
|:------:|---------|----------|----------|----------|
| 高 | `branch-outdated` | behind > 0 | 您的分支落后 {n} 个提交 | gitm update |
| 高 | `conflict-risk` | 检测到冲突风险 | 检测到 {n} 个文件可能冲突 | gitm check --combine |
| 高 | `important-unpushed` | ahead > 0 && 有重要提交 | 有重要提交未推送 | git push |
| 中 | `ready-to-combine` | 有未推送 && 无冲突 | 可以合并到测试环境 | gitm combine -d |
| 中 | `long-no-sync` | 未同步 > 7 天 | 已 {n} 天未同步上游代码 | gitm update |
| 中 | `many-unpushed` | ahead > 3 | 有 {n} 个提交未推送 | git push |
| 低 | `branch-cleanup` | mergedBranches > 5 | 有 {n} 个已合并分支可清理 | gitm cleanbranch |
| 低 | `orphan-worktree` | 有孤立工作区 | 有 {n} 个无效工作区 | gitm worktree prune |

#### 5.3.3 上下文感知

**GitContext 数据结构**：

```typescript
interface GitContext {
    // 分支信息
    branch: {
        current: string           // 当前分支名
        type: 'feature' | 'bugfix' | 'support' | 'main' | 'other'
        ahead: number             // 领先远程的提交数
        behind: number            // 落后远程的提交数
        upstream: string          // 上游分支
        lastSync: Date            // 最后同步时间
    }

    // 文件状态
    files: {
        staged: string[]          // 已暂存
        unstaged: string[]        // 未暂存
        untracked: string[]       // 未跟踪
        conflicts: string[]       // 冲突文件
    }

    // 提交信息
    commits: {
        recent: Commit[]          // 最近提交
        unpushed: Commit[]        // 未推送提交
        important: Commit[]       // 重要提交（根据规则判断）
    }

    // 仓库状态
    repo: {
        isClean: boolean          // 是否干净
        hasConflict: boolean      // 是否有冲突
        conflictRisk: 'none' | 'low' | 'medium' | 'high'  // 冲突风险
        mergedBranches: string[]  // 已合并分支
    }

    // 项目上下文
    project: {
        type: 'node' | 'java' | 'python' | 'go' | 'unknown'
        hasPackageJson: boolean
        hasTest: boolean
    }
}
```
